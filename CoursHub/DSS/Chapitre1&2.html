1.1

Contexte et Problématique

POURQUOI CE COURS ?
Le Web génère des données hétérogènes, évolutives, hiérarchiques. Les bases de données relationnelles (SQL) ne suffisent plus à les gérer efficacement. Il faut des modèles plus flexibles — c'est le domaine des données semi-structurées.
instagram_post.json — Cas concret : pourquoi pas SQL ? 
{
  "user": "sara_tech",           // ✓ toujours présent
  "date": "2025-10-21",          // ✓ toujours présent
  "text": "Learning XML is fun!",   // ? optionnel — certains posts ont juste une image
  "hashtags": ["#Data", "#XML", "#Tech"],  // ? nombre variable : 0 à 30 !
  "likes": 125,
  "media": {
    "type": "image",             // ? parfois vidéo, parfois carousel
    "url": "photo.jpg"
  },
  "comments": [                    // ? nombre illimité + réponses imbriquées
    {
      "user": "ali",
      "text": "Great post!",
      "replies": [{ "user": "sara_tech", "text": "Thanks 😊" }]
    }
  ]
}
PROBLÈME SQL
En SQL, chaque ligne d'une table a exactement le même nombre de colonnes. Comment créer une table posts si chaque post peut avoir 0 à 30 hashtags, 0 à ∞ commentaires et plusieurs types de médias ? C'est impossible proprement avec un schéma fixe.

Problème Tentative naïve en SQL — Pourquoi ça échoue

+
posts_naive.sql — Le problème des colonnes variables 
-- Tentative naïve : colonnes fixes
CREATE TABLE posts (
  id          INT PRIMARY KEY,
  user        VARCHAR(50),
  date        DATE,
  text        TEXT,        -- NULL si pas de texte
  hashtag1    VARCHAR(50), -- Et s'il y en a 20 ?
  hashtag2    VARCHAR(50), -- On crée 30 colonnes ? 99% NULL !
  hashtag3    VARCHAR(50),
  -- ... jusqu'à hashtag30 ?
  comment1    TEXT,        -- Idem pour les commentaires
  comment2    TEXT,        -- Et les réponses aux commentaires ?
  media_type  VARCHAR(10)  -- 'image' | 'video' | 'carousel'
);

-- Problèmes :
-- ❌ 90% des cellules seraient NULL (gaspillage)
-- ❌ Impossible de stocker des commentaires imbriqués
-- ❌ ALTER TABLE nécessaire pour chaque nouvelle feature
-- ❌ Jointures complexes pour les données hiérarchiques
SOLUTION
Utiliser un format semi-structuré (XML ou JSON) qui autorise une structure flexible et hiérarchique. Les balises/clés portent le sens des données, et chaque document peut avoir sa propre structure.

Concept Qu'est-ce qu'une donnée semi-structurée ?

+
🏷️

Auto-descriptive

Les données portent leur propre description. Les balises XML ou les clés JSON expliquent directement ce que signifie chaque valeur, sans schéma externe obligatoire.

🌳

Hiérarchique

Modèle en arbre (nœuds, feuilles). Permet l'imbrication naturelle : un livre contient des chapitres qui contiennent des sections — impossible avec un tableau plat.

🔀

Flexible

Pas de schéma obligatoire défini à l'avance. On peut ajouter de nouveaux champs sans migration de base de données. Idéal pour les données évolutives.

📡

Portable

Format texte universel, lisible par n'importe quel langage (Java, Python, JS, PHP...) et n'importe quelle plateforme. Le format standard d'échange sur le Web.

1.2

Les 3 Types de Données

Toutes les données ne se ressemblent pas. On distingue trois grandes familles selon leur degré de structure. Cliquez sur chaque carte pour explorer les détails.

Structurées
🗄️ Structurées

Schéma fixe défini à l'avance. Chaque enregistrement a exactement les mêmes champs.

Cohérence garantie Requêtes SQL puissantes Rigide, difficile à évoluer Pas adapté aux données hétérogènes
Ex : Table SQL, CSV, Excel
FOCUS DU COURS ★
🌳 Semi-Structurées

Structure flexible et hiérarchique. Schéma optionnel. Chaque document peut avoir sa propre organisation.

Flexibilité maximale Auto-descriptif Idéal pour le Web Requêtes plus complexes (XPath, XQuery)
Ex : XML, JSON, YAML, HTML
Non Structurées
📄 Non Structurées

Aucun schéma formel. Le contenu est du texte libre, des images, des vidéos — pas d'organisation prédéfinie.

Liberté totale de format Difficile à analyser automatiquement Besoin d'IA / NLP pour traiter
Ex : PDF, Images, Vidéos, emails bruts

STATISTIQUE CLÉ
Selon les estimations, plus de 80% des données générées dans le monde sont semi-structurées ou non-structurées (réseaux sociaux, emails, fichiers multimédia, logs applicatifs). Le SQL traditionnel ne peut pas tout gérer.
1.3

Limites des Bases de Données Relationnelles

CONTEXTE
Les SGBD relationnels (SQL) ont dominé l'informatique depuis les années 70. Excellents pour des données stables et uniformes, ils montrent leurs limites face aux besoins du Web moderne.

🔒 Rigidité du Schéma — ALTER TABLE, le cauchemar

+

Ajouter un champ couleur_yeux à une table de 1 million d'utilisateurs :

SQL — Opération lourde 
-- SQL : modification lourde et risquée
ALTER TABLE utilisateurs
  ADD COLUMN couleur_yeux VARCHAR(20);

-- Conséquences :
-- ❌ Verrouillage de la table pendant l'opération
-- ❌ 1 000 000 lignes avec couleur_yeux = NULL
-- ❌ Temps d'arrêt potentiel du service
-- ❌ Migration de données nécessaire

Même chose en XML :

XML — Simple et non-disruptif 
<!-- Anciens docs : inchangés -->
<utilisateur>
  <nom>Dupont</nom>
</utilisateur>

<!-- Nouveau doc : champ ajouté librement -->
<utilisateur>
  <nom>Martin</nom>
  <couleur_yeux>bleus</couleur_yeux>
</utilisateur>
<!-- ✓ 0 impact sur les anciens documents -->

🛒 Données Hétérogènes — Le problème e-commerce

+
SCÉNARIO
Un site e-commerce vend des livres ET des vêtements. Leurs attributs sont complètement différents.

// SQL — Solution bancale

-- Table produits avec colonnes NULL
CREATE TABLE produits (
  id          INT,
  nom         VARCHAR(100),
  prix        DECIMAL,
  -- Attributs livres (NULL pour vêtements)
  auteur      VARCHAR(100), -- NULL
  isbn        VARCHAR(20),  -- NULL
  nb_pages    INT,          -- NULL
  -- Attributs vêtements (NULL pour livres)
  taille      VARCHAR(10),  -- NULL
  couleur     VARCHAR(30),  -- NULL
  matiere     VARCHAR(50)   -- NULL
)
-- Résultat : 50% de NULL, structure confuse !

// XML — Chaque produit = sa structure

<livre>
  <nom>XML Masterclass</nom>
  <prix>45.00</prix>
  <auteur>Jean Dupont</auteur>
  <isbn>978-3-16-148410-0</isbn>
</livre>

<vetement>
  <nom>T-shirt Premium</nom>
  <prix>29.99</prix>
  <taille>M</taille>
  <couleur>noir</couleur>
</vetement>

🔗 Échange de Données — L'interopérabilité

+

❌ Export SQL

Format propriétaire dépendant du SGBD. Un export MySQL n'est pas directement lisible par Oracle ou SQL Server. Nécessite des outils de conversion.

✅ Export XML / JSON

Format texte universel. Lisible par n'importe quel langage (Java, Python, JavaScript, PHP...) et n'importe quelle plateforme. C'est le standard du Web.

INTEROPÉRABILITÉ XML
XML a été conçu comme l'Intermédiaire Universel des données. Java parle à PHP, Python parle à .NET, tous via XML. C'est pourquoi les APIs Web, SOAP, RSS et la configuration logicielle utilisent XML ou JSON.

🌐 Données Hiérarchiques — L'impossibilité des arbres en SQL

+
catalogue.xml — Données naturellement hiérarchiques 
<?xml version="1.0" encoding="UTF-8"?>
<bibliotheque>
  <rayon genre="roman">
    <livre>
      <titre>Les Misérables</titre>
      <auteur>
        <prenom>Victor</prenom>
        <nom>Hugo</nom>
        <nationalite>Française</nationalite>
      </auteur>
      <chapitres>
        <chapitre num="1">Fantine</chapitre>
        <chapitre num="2">Cosette</chapitre>
      </chapitres>
    </livre>
  </rayon>
</bibliotheque>

<!-- En SQL : il faudrait 4 tables (bibliotheque, rayon, livre, chapitre)
     + 3 JOIN pour reconstituer cette hiérarchie simple ! -->
2.1

Documents et Hyperdocuments Multimédias

CONTEXTE
Avant d'étudier XML, il faut comprendre ce qu'est un document numérique, comment les documents sont devenus des hyperdocuments liés entre eux, et pourquoi le Web a eu besoin d'un format de données structuré universel.
DÉFINITION — DOCUMENT
Un document est une entité informationnelle destinée à être lue, consultée ou traitée par des humains ou des machines. Il contient de l'information structurée avec un sens précis.
📋

Types de Documents

  • Rapports et articles scientifiques
  • Contrats et documents légaux
  • Catalogues produits
  • Pages web (HTML)
  • Fichiers de configuration
🏗️

Structure d'un Document

  • Contenu : l'information elle-même
  • Structure : organisation logique (titres, sections)
  • Présentation : rendu visuel (police, couleur)
  • Métadonnées : auteur, date, mots-clés
💡

Principe de Séparation

XML sépare rigoureusement le contenu (ce qu'on dit) de la présentation (comment on l'affiche). Un même document XML peut être rendu en HTML, PDF, ou traité par une machine.

DÉFINITION — HYPERDOCUMENT
Un hyperdocument est un document enrichi de liens (hyperliens) permettant une navigation non-linéaire entre différents contenus. C'est la base du Web — HTML est un langage d'hyperdocument.
mooc.xml — Exemple : MOOC avec navigation non-linéaire 
<cours id="DSS101">
  <titre>Données Semi-Structurées</titre>
  <module id="m1">
    <video url="intro.mp4"/>
    <lien href="quiz.xml">→ Quiz d'auto-évaluation</lien>
    <lien href="ressources.xml">→ Ressources complémentaires</lien>
    <lien href="forum.xml">→ Forum de discussion</lien>
  </module>
</cours>

✅ Avantages

  • Navigation personnalisée
  • Interactivité et engagement
  • Collaboration entre utilisateurs
  • Mise à jour centralisée

📚 Technologies associées

  • XLink / XPointer — liens XML
  • HTML — hyperdocument web
  • URI / URL — identification
  • RDF — liens sémantiques
MULTIMÉDIA
Un document multimédia combine plusieurs types de médias : texte, images, sons, vidéos, animations. XML permet de structurer et de référencer ces différents types de contenus.
📝

Texte

Le contenu principal, structuré avec des balises XML. Peut inclure des métadonnées comme la langue, l'auteur, la date.

🖼️

Images

Référencées via des attributs XML. SVG (Scalable Vector Graphics) est lui-même un langage XML pour les graphiques vectoriels.

🎵

Audio / Vidéo

Métadonnées stockées en XML (titre, durée, codec). Les playlists et catalogues médias sont souvent en XML.

🎯

Exemples concrets

  • Podcasts : flux RSS en XML
  • YouTube Data API : XML/JSON
  • SVG : graphiques XML natifs
  • EPUB : livres en XML + HTML
PRINCIPE FONDAMENTAL
XML implémente rigoureusement la séparation entre le Contenu et la Présentation. C'est l'un de ses apports les plus importants par rapport à HTML.

📄 Le Contenu (XML)

Ce que les données signifient — indépendant de tout affichage.

<article>
  <titre>XML Fondamental</titre>
  <auteur>Jean Dupont</auteur>
  <texte>XML est...</texte>
</article>
<!-- Pas de couleur, taille, police ici ! -->

🎨 La Présentation (XSL/CSS)

Comment afficher — séparé des données.

/* CSS ou XSL */
article titre {
  font-size: 2rem;
  color: #333;
  font-weight: bold;
}
article auteur {
  font-style: italic;
  color: gray;
}
/* Le même XML = 10 rendus différents ! */
AVANTAGE CLÉ
Un seul fichier XML peut être transformé en page web (via XSLT), en PDF (via XSL-FO), en données JSON (via transformation), ou traité directement par une machine. C'est la puissance de la séparation contenu/présentation.
2.2

Évolution : SGML → HTML → XML

FILIATION
XML n'a pas été inventé de nulle part. Il descend directement de SGML (1986), qui avait engendré HTML (1991). XML (1998) est une simplification de SGML adaptée au Web.
1986 — ISO 8879

SGML — Standard Generalized Markup Language

Philosophie : "Je peux décrire n'importe quelle structure de document"

📌 Principe

Métalangage de balisage standardisé (ISO). Permet de définir des langages de balisage personnalisés. HTML et XML en sont des sous-ensembles.

⚙️ Usages

  • Documentation technique (Boeing, Airbus)
  • Édition électronique professionnelle
  • Standard industriel pour grands groupes

❌ Problèmes

  • Trop complexe pour un usage quotidien
  • Outils très coûteux
  • Difficile à implémenter dans les navigateurs
SGML — Exemple de déclaration 
<!-- Définition d'un type de document en SGML -->
<!DOCTYPE livre [
  <!ELEMENT livre (titre, auteur, chapitre+)>
  <!ELEMENT titre (#PCDATA)>
  <!ELEMENT auteur (#PCDATA)>
  <!ELEMENT chapitre (para*)>
]>
<!-- HTML et XML utilisent une syntaxe DTD similaire -->
1991 — Tim Berners-Lee / W3C

HTML — HyperText Markup Language

Philosophie : "Je décris comment afficher l'information"

HTML — Présentation, balises fixes 
<html>
  <body>
    <h1>Java et Internet</h1>
    <p>Par Gilles Roussel</p>
    <b>Éditeur :</b> Vuibert
    <br/>600 pages
  </body>
</html>

<!-- <h1>, <p>, <b> = balises prédéfinies,
     fixes, impossibles à personnaliser -->
LIMITES HTML
  • ❌ Balises fixes — impossible d'ajouter <livre>
  • ❌ Mélange contenu + présentation
  • ❌ Pas de validation de structure
  • ❌ Pas adapté à l'échange de données
  • ❌ Pas extensible
ATOUTS HTML
  • ✓ Simple à apprendre
  • ✓ Supporté par tous les navigateurs
  • ✓ Idéal pour l'affichage web
  • ✓ Liens hypertextes natifs
1998 — W3C (Recommandation Officielle)

XML — eXtensible Markup Language

Philosophie : "I provide meaning" — Je décris ce que les données signifient

XML — Sémantique, balises libres 
<?xml version="1.0" encoding="UTF-8"?>
<livre>
  <titre>Java et Internet</titre>
  <auteur>
    <prenom>Gilles</prenom>
    <nom>Roussel</nom>
  </auteur>
  <editeur>Vuibert</editeur>
  <nb_pages>600</nb_pages>
</livre>

<!-- Balises définies par l'utilisateur !
     La structure = le sens -->
APPORTS XML
  • ✓ Balises libres — vous définissez votre langage
  • ✓ Séparation contenu / présentation
  • ✓ Validation via DTD ou XML Schema
  • ✓ Extensible sans limite
  • ✓ Interopérable — standard W3C/ISO
  • ✓ Lisible par humains et machines
2000+ — Aujourd'hui

JSON, YAML, GraphQL — La famille continue

XML a inspiré d'autres formats semi-structurés plus légers pour les APIs web.

📦

JSON

Basé sur la syntaxe JavaScript. Plus léger que XML pour les APIs REST. Moins de verbosité, mais sans validation native ni namespaces.

📋

YAML

Lisible par les humains. Utilisé pour la configuration (Docker Compose, Kubernetes, CI/CD pipelines).

🔗

GraphQL

Langage de requête pour APIs. Permet de récupérer exactement les données nécessaires, ni plus ni moins.

Domaines d'Application et Écosystème

XML EST PARTOUT
XML est le format de données structurées le plus répandu dans l'industrie logicielle. Même si JSON a pris sa place dans les APIs REST, XML reste incontournable dans de nombreux domaines.
🔄

Échange de Données

Protocoles SOAP (Web Services), flux RSS/Atom (actualités, podcasts), APIs B2B entre entreprises.

🗄️

Stockage Natif XML

Bases XML natives : eXist-db, MarkLogic, BaseX. Stockent et requêtent du XML directement avec XQuery.

⚙️

Configuration Logicielle

web.xml (Java EE), pom.xml (Maven), AndroidManifest.xml, Spring Framework, configurations Tomcat...

📄

Formats de Documents

OOXML (Word/Excel/PowerPoint), EPUB (livres numériques), DocBook (documentation technique), SVG (graphiques vectoriels).

🤖

Machine Learning / IA

Datasets annotés pour la vision (PASCAL VOC en XML), corpus NLP, configuration de modèles TensorFlow, annotations d'images.

📚

Métadonnées et Web Sémantique

Dublin Core (bibliothèques), RDF/XML (Web sémantique), OWL (ontologies), ATOM (syndication de contenu).

AVANTAGES DE XML EN RÉSUMÉ
Interopérabilité : échange entre Java, .NET, PHP, Python  ·  Neutralité : indépendant des langages et SGBD  ·  Standardisation : W3C / ISO reconnu mondialement  ·  Extensibilité : vous définissez vos propres balises  ·  Validation : DTD et XML Schema garantissent la qualité des données

🎯 Points Clés à Retenir

  • 1
    3 types de données — Structurées (SQL), Semi-Structurées (XML/JSON) ← focus du cours, Non-Structurées (PDF, images).
  • 2
    Limites SQL — Rigidité du schéma, données hétérogènes difficiles, échange inter-systèmes compliqué, hiérarchies impossibles proprement.
  • 3
    Évolution des langages — SGML (1986, complexe) → HTML (1991, affichage) → XML (1998, données + extensible).

  • 5

  • 6
    Écosystème riche — XPath, XQuery, XSLT, DOM, SAX, DTD, XML Schema, XLink... XML est bien plus qu'un simple format de fichier.
